# I18n

You are auditing this codebase for internationalization (i18n) and localization (l10n) issues. Your job is to find every place where the application would break, look wrong, or confuse users in a non-English locale or right-to-left language. i18n bugs are invisible to monolingual developers — that's why they ship to production.

**Scope:** $ARGUMENTS

---

## Phase 1: Hardcoded Strings

The most common i18n bug: text that can't be translated because it's hardcoded.

### Scan for Untranslated Text

Search ALL user-facing code for hardcoded strings. This includes:

1. **UI labels and messages** — button text, menu items, headers, descriptions, placeholders
2. **Error messages** — validation messages, API error displays, empty states
3. **Notifications** — push notifications, in-app alerts, toasts, confirmations
4. **Emails and SMS** — any text sent to users
5. **PDF/report generation** — headers, labels, footers in generated documents
6. **Logging that reaches users** — error pages, status messages

### What to Flag

```
[HARDCODED] file:line — "the hardcoded string" — context: [where it appears in UI]
```

### What NOT to Flag

- Log messages (internal, not user-facing)
- Developer-facing error codes (`ERR_NOT_FOUND`)
- Technical identifiers (field names, routes, keys)
- Test data
- Comments

### String Extraction Readiness

For each hardcoded string found:
- Is there an existing translation system? (i18n library, ARB files, resource bundles, gettext)
- Does the string require interpolation (variables inside the string)?
- Does the string require pluralization?
- Is the string context-dependent (same word means different things in different places)?

## Phase 2: Layout & Direction

### RTL (Right-to-Left) Issues

For languages like Arabic, Hebrew, Persian, and Urdu, the entire layout mirrors. Check for:

- [ ] **Physical direction in code.** `left`/`right` in padding, margin, alignment, positioning → should be `start`/`end` or logical equivalents.
  - CSS: `margin-left` → `margin-inline-start`, `text-align: left` → `text-align: start`
  - Flutter: `EdgeInsets.only(left:)` → `EdgeInsetsDirectional.only(start:)`
  - React Native: `paddingLeft` → `paddingStart`

- [ ] **Icons and directional graphics.** Arrows, navigation icons, progress indicators — these need to mirror in RTL. (Back arrow should point right in RTL.)

- [ ] **Text alignment.** Hardcoded `text-align: left` breaks RTL. Use `start` instead.

- [ ] **Positional words in code.** Check for `Alignment.centerLeft`/`centerRight` → should use directional equivalents.

- [ ] **Swipe gestures.** Swipe-left-to-delete should become swipe-right-to-delete in RTL.

- [ ] **Progress and sequences.** Timelines, progress bars, step indicators — these should flow in reading direction.

- [ ] **Bidirectional text.** When RTL and LTR text mix (e.g., Arabic sentence with an English brand name), is the rendering correct? Are Unicode bidi markers used where needed?

### Responsive Layout for Long Text

Different languages have very different text lengths:

| Language | Relative to English |
|----------|-------------------|
| German | 30% longer |
| Finnish | 30-40% longer |
| Japanese | 50% shorter |
| Arabic | 20-25% longer |
| Chinese | 50% shorter |

Check for:
- [ ] **Fixed-width containers.** Will text overflow or truncate in longer languages?
- [ ] **Hardcoded heights.** Will multi-line text in German break a fixed-height box?
- [ ] **Truncation.** Is truncated text still meaningful? Does `...` appear at the right end?
- [ ] **Button text.** Will translated button labels fit? (English "OK" → German "Einverstanden")
- [ ] **Table columns.** Will column headers and data fit in all languages?

## Phase 3: Data Formatting

### Dates and Times

- [ ] **Date format varies by locale.** MM/DD/YYYY (US), DD/MM/YYYY (EU), YYYY/MM/DD (Asia). NEVER hardcode format.
- [ ] **Using locale-aware formatting.** `DateFormat.yMd(locale)` not string concatenation.
- [ ] **Relative dates are locale-aware.** "3 days ago" must be translated, not assembled from parts.
- [ ] **Time zones handled.** Storing UTC, displaying local. Not the other way around.
- [ ] **Calendar system.** Some locales use non-Gregorian calendars (Hijri, Buddhist). Is this relevant?
- [ ] **First day of week.** Sunday (US), Monday (EU/most of world), Saturday (Middle East).

### Numbers and Currency

- [ ] **Decimal separator.** Period (US/UK: 1,234.56) vs comma (EU: 1.234,56). NEVER hardcode.
- [ ] **Thousands separator.** Comma, period, space, or nothing — varies by locale.
- [ ] **Currency formatting.** Symbol position ($100 vs 100€ vs 100 €), symbol vs code ($ vs USD), number of decimal places (JPY has 0, most have 2, some have 3).
- [ ] **Currency stored correctly.** Integers (cents), not floats. Currency code stored alongside amount.
- [ ] **Percentage formatting.** Some locales put space before % sign.
- [ ] **Phone number formatting.** International format with country code, local formatting varies.

### Text Processing

- [ ] **Sorting/collation is locale-aware.** German ä sorts differently than English a. Spanish ch was historically a single letter.
- [ ] **String comparison is locale-aware.** Case-insensitive comparison varies (Turkish İ/i problem).
- [ ] **Text length validation accounts for encoding.** A Japanese character is 1 character but 3 bytes in UTF-8. Max lengths should be in characters, not bytes — or explicitly documented.
- [ ] **Line breaking rules.** CJK languages break differently than Western languages. Thai has no spaces between words.

### Pluralization

- [ ] **Plural forms handled by the i18n system, not by code.** Not `count == 1 ? "item" : "items"` — this breaks for languages with multiple plural forms.
  - English: 2 forms (1, other)
  - Arabic: 6 forms (zero, one, two, few, many, other)
  - Chinese/Japanese: 1 form (no plural distinction)
- [ ] **Zero case handled.** "0 items" vs "No items" — both valid, but must be translatable.
- [ ] **Gender agreement.** Some languages require different text based on the gender of the subject. Is this accounted for?

### Severity

- **HIGH** — Text displayed in wrong language, data displayed incorrectly (wrong date format, wrong number), or layout completely broken in RTL.
- **MEDIUM** — Text truncated in some languages, minor layout issues, missing pluralization support.
- **LOW** — Best practice improvements, potential issues that may not affect currently supported locales.

## Hard Rules

1. **NEVER assume English text patterns are universal.** "First name Last name" is not universal (some cultures put family name first). "Street, City, State, ZIP" is US-only. "Dear Mr./Ms." requires gendered addressing that not all cultures use.

2. **NEVER concatenate strings to build sentences.** `"Welcome, " + name + "!"` breaks in languages where the name goes in the middle or the sentence structure differs. Use parameterized translation strings: `t('welcome', {name})`.

3. **NEVER hardcode date, number, or currency formats.** Always use the platform's locale-aware formatting APIs. What looks right in your locale looks wrong in half the world.

4. **NEVER assume text direction.** Every `left`/`right` in layout code is a potential RTL bug. Use `start`/`end` logical properties everywhere.

5. **NEVER implement plural logic in application code.** Use the i18n framework's plural system (ICU MessageFormat, CLDR plural rules, or framework equivalent). Application-level `if (count == 1)` breaks for the majority of the world's languages.

6. **NEVER assume character encoding.** Use UTF-8 everywhere. Test with non-ASCII input: accented characters (é, ñ), CJK characters (日本語), emoji (🎉), RTL characters (العربية).
